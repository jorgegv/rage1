# GENERIC-SPRITE-LIB-REFACTOR: Abstracting SP1 behind a sprite drawing layer

## 1. Goal

Decouple the RAGE1 engine from the z88dk SP1 sprite library so that SP1
can be replaced by an alternative sprite/tile drawing library with less
memory overhead, without changing any engine logic or game data files.

The abstraction must:
- Introduce zero (or near-zero) runtime overhead on the current SP1
  backend — wrapper functions will mostly be thin inline calls or macros.
- Keep the same build pipeline (`datagen.pl` → C source → z88dk compile).
- Not break 48K or 128K builds, banked code, or dataset/codeset
  management.
- Be selectable at build time via a compile-time flag (e.g.
  `BUILD_FEATURE_SPRITE_ENGINE_SP1` vs `BUILD_FEATURE_SPRITE_ENGINE_ALT`).

### 2.6 Banked code considerations

The following SP1-using functions run from main (lower) memory, NOT
from banked code:

- All sprite drawing functions (`sp1_MoveSprPix`, `sp1_MoveSprAbs`,
  `sp1_PrintAtInv`, `sp1_ClearRectInv`, `sp1_UpdateNow`, etc.)
- `sprite_allocate()`, `sprite_free()`, `sprite_move_offscreen()`,
  `sprite_set_color()` (all in `engine/src/sprite.c`)
- `hero_draw()`, `hero_reset_position()` (in `engine/src/hero.c`)
- `enemy_reset_position_all()`, `enemy_redraw_all()` (in
  `engine/src/enemy.c`)
- `bullet_redraw_all()`, `bullet_init_sprites()` (in
  `engine/src/bullet.c`)
- `btile_draw()`, `btile_draw_frame()`, `btile_remove()` (in
  `engine/src/btile.c`)
- `map_draw_screen()`, `map_allocate_sprites()`, `map_free_sprites()`
  (in `engine/src/map.c`)

The banked code (`engine/banked_code/common/`) handles only animation
and movement _calculations_ — it does NOT call any SP1 drawing functions
directly.  It computes positions and sets `F_ENEMY_NEEDS_REDRAW` /
`F_BULLET_NEEDS_REDRAW` / `F_LOOP_REDRAW_HERO` flags, and the actual
redraw calls happen from main memory afterwards.

This means: **the abstraction layer code can reside entirely in main
memory, alongside the current SP1 calls.  No banked-code changes are
needed for the abstraction itself.**

## 3. Abstraction layer design

### 3.1 New header: `engine/include/rage1/gfx.h`

This is the single engine-facing header for all sprite/tile drawing
operations.  Engine files will `#include "rage1/gfx.h"` instead of
`<games/sp1.h>`.

`gfx.h` itself contains **no backend-specific code**.  It only:
1. Documents the generic API (as comments)
2. Includes the backend-specific header based on the build feature flag
3. Declares prototypes for the few real functions (see section 3.3)

```c
#ifndef _GFX_H
#define _GFX_H

#include <stdint.h>
#include "features.h"

/////////////////////////////////////////////////////////////////////
// Generic GFX API
//
// All engine code uses these names.  Each backend-specific header
// (gfx_sp1.h, gfx_alt.h, ...) must provide:
//
//   Types (as typedefs):
//     gfx_sprite_t       - opaque handle to a hardware sprite
//     gfx_rect_t         - rectangle (row, col, width, height)
//     gfx_print_ctx_t    - print-string context
//
//   Constants (as #defines):
//     GFX_CLEAR_TILE     - flag: clear tiles in a rect
//     GFX_CLEAR_COLOUR   - flag: clear colour in a rect
//     GFX_PSS_INVALIDATE - flag: invalidate on print
//     GFX_PRINT_CTX_INIT(area, attr) - static initializer for print ctx
//
//   Macros (mapping to library functions):
//     gfx_invalidate(rect)
//     gfx_update()
//     gfx_sprite_destroy(s)
//     gfx_sprite_set_threshold(s, xt, yt)
//     gfx_sprite_move_pixel(s, clip, frame, x, y)
//     gfx_sprite_move_cell(s, clip, frame, row, col)
//     gfx_sprite_get_row(s)
//     gfx_sprite_get_col(s)
//     gfx_sprite_get_width(s)
//     gfx_sprite_get_height(s)
//     gfx_tile_put(row, col, attr, tile)
//     gfx_tile_register(index, graphic)
//     gfx_clear_rect(rect, attr, ch, flags)
//     gfx_print_set_pos(ctx, row, col)
//     gfx_print_string(ctx, str)
//
/////////////////////////////////////////////////////////////////////

// Include the backend-specific header
#ifdef BUILD_FEATURE_SPRITE_ENGINE_SP1
    #include "rage1/gfx_sp1.h"
#endif

#ifdef BUILD_FEATURE_SPRITE_ENGINE_ALT
    #include "rage1/gfx_alt.h"
#endif

// Real functions (multi-step, backend-specific body in .c files)
void gfx_init( uint8_t bg_attr, uint8_t bg_char );
gfx_sprite_t *gfx_sprite_create( uint8_t rows, uint8_t cols );
void gfx_sprite_set_color( gfx_sprite_t *s, uint8_t color );

#endif // _GFX_H
```

### 3.2 Backend-specific headers

Each sprite engine backend gets its own header file in
`engine/include/rage1/`.  These files are **never included directly**
by engine code — they are included only by `gfx.h` via the conditional
`#include` shown above.

Each backend header must provide the full set of typedefs, constants,
and macros documented in `gfx.h`.

**Critical design constraint:** On the Z80, every function call has
significant overhead (CALL/RET, stack frame, parameter passing).  The
`gfx_*` mappings must therefore be implemented **entirely as macros**,
not as wrapper functions.  This gives exactly zero runtime overhead —
the preprocessor replaces each `gfx_*` call with the direct library
call at compile time.

**This rule applies to ALL backends, not just SP1.**  When a new sprite
library is integrated, its `gfx_*.h` header must also define macros
that expand directly to that library's functions.  No backend should
ever use wrapper functions for the `gfx_*` API.

The only exceptions are `gfx_init()`, `gfx_sprite_create()`, and
`gfx_sprite_set_color()`, which involve multi-step logic that varies per
backend.  These are real functions declared in `gfx.h` and defined in
existing `.c` files (see section 6.5).

#### `engine/include/rage1/gfx_sp1.h` — SP1 backend

```c
#ifndef _GFX_SP1_H
#define _GFX_SP1_H

#include <games/sp1.h>

//--- Types ---
typedef struct sp1_ss    gfx_sprite_t;
typedef struct sp1_Rect  gfx_rect_t;
typedef struct sp1_pss   gfx_print_ctx_t;

//--- Constants ---
#define GFX_CLEAR_TILE         SP1_RFLAG_TILE
#define GFX_CLEAR_COLOUR       SP1_RFLAG_COLOUR
#define GFX_PSS_INVALIDATE     SP1_PSSFLAG_INVALIDATE
#define GFX_PRINT_CTX_INIT(area, attr) \
    { &(area), GFX_PSS_INVALIDATE, 0, 0, 0, (attr), 0, 0 }

//--- Initialization ---
// gfx_init() is a real function (multi-step), defined in sp1engine.c
#define gfx_invalidate(rect)                   sp1_Invalidate(rect)
#define gfx_update()                           sp1_UpdateNow()

//--- Sprite lifecycle ---
// gfx_sprite_create() is a real function (multi-step), defined in sprite.c
#define gfx_sprite_destroy(s)                  sp1_DeleteSpr(s)
// gfx_sprite_set_color() is a real function (multi-step), defined in sprite.c
#define gfx_sprite_set_threshold(s,xt,yt) \
    do { (s)->xthresh = (xt); (s)->ythresh = (yt); } while(0)

//--- Sprite movement ---
#define gfx_sprite_move_pixel(s,clip,fr,x,y)   sp1_MoveSprPix((s),(clip),(fr),(x),(y))
#define gfx_sprite_move_cell(s,clip,fr,r,c)    sp1_MoveSprAbs((s),(clip),(fr),(r),(c),0,0)

//--- Sprite query ---
#define gfx_sprite_get_row(s)                  ((s)->row)
#define gfx_sprite_get_col(s)                  ((s)->col)
#define gfx_sprite_get_width(s)                ((s)->width)
#define gfx_sprite_get_height(s)               ((s)->height)

//--- Tile drawing ---
#define gfx_tile_put(r,c,attr,tile)            sp1_PrintAtInv((r),(c),(attr),(tile))
#define gfx_tile_register(idx,gfx)             sp1_TileEntry((idx),(gfx))

//--- Rectangle operations ---
#define gfx_clear_rect(rect,attr,ch,flags)     sp1_ClearRectInv((rect),(attr),(ch),(flags))

//--- Text printing ---
#define gfx_print_set_pos(ctx,r,c)             sp1_SetPrintPos((ctx),(r),(c))
#define gfx_print_string(ctx,str)              sp1_PrintString((ctx),(str))

#endif // _GFX_SP1_H
```

#### `engine/include/rage1/gfx_alt.h` — placeholder for alternative backend

When a new backend is added, create this file with the same structure:
library include, typedefs, constants, and macros — all mapping to that
library's API.

```c
#ifndef _GFX_ALT_H
#define _GFX_ALT_H

#include "alt_sprite_lib.h"

//--- Types ---
typedef struct alt_sprite   gfx_sprite_t;
typedef struct alt_rect     gfx_rect_t;
typedef struct alt_print    gfx_print_ctx_t;

//--- Constants ---
#define GFX_CLEAR_TILE         ALT_CLEAR_TILE
#define GFX_CLEAR_COLOUR       ALT_CLEAR_COLOUR
#define GFX_PSS_INVALIDATE     ALT_PSS_INVALIDATE
#define GFX_PRINT_CTX_INIT(area, attr)  /* alt-library-specific init */

//--- Macros (all map directly to alt library functions) ---
#define gfx_invalidate(rect)                   alt_invalidate(rect)
#define gfx_update()                           alt_update()
#define gfx_sprite_destroy(s)                  alt_destroy_sprite(s)
#define gfx_sprite_set_threshold(s,xt,yt)      alt_set_threshold((s),(xt),(yt))
#define gfx_sprite_move_pixel(s,clip,fr,x,y)   alt_move_sprite_px((s),(clip),(fr),(x),(y))
#define gfx_sprite_move_cell(s,clip,fr,r,c)    alt_move_sprite_cell((s),(clip),(fr),(r),(c))
#define gfx_sprite_get_row(s)                  alt_get_row(s)
#define gfx_sprite_get_col(s)                  alt_get_col(s)
#define gfx_sprite_get_width(s)                alt_get_width(s)
#define gfx_sprite_get_height(s)               alt_get_height(s)
#define gfx_tile_put(r,c,attr,tile)            alt_put_tile((r),(c),(attr),(tile))
#define gfx_tile_register(idx,gfx)             alt_register_tile((idx),(gfx))
#define gfx_clear_rect(rect,attr,ch,flags)     alt_clear_rect((rect),(attr),(ch),(flags))
#define gfx_print_set_pos(ctx,r,c)             alt_set_print_pos((ctx),(r),(c))
#define gfx_print_string(ctx,str)              alt_print_string((ctx),(str))

#endif // _GFX_ALT_H
```

**There is no `gfx.c` or `gfx_alt.c` file.**  The entire translation
layer lives in the backend header files as macros.

### 3.3 Backend selection

Backend selection is controlled by a single `BUILD_FEATURE_*` flag (set
in `features.h` by `datagen.pl`).  The `gfx.h` header uses this flag
to `#include` the correct backend header — see the code in section 3.1.

Since everything is macros and typedefs, **no Makefile changes are
needed for backend selection**.  The build feature flag alone controls
which backend is active.

`datagen.pl` should auto-enable `BUILD_FEATURE_SPRITE_ENGINE_SP1` as the
default, so existing games build unchanged.

## 6. Considerations and risks

### 6.1 Memory overhead

The abstraction layer is implemented entirely as macros in `gfx.h`,
so it adds **zero** runtime overhead and **zero** code size increase
for any backend.  Each `gfx_*` macro expands directly to the
corresponding library function call at compile time — there are no
wrapper functions, no extra CALL/RET pairs, no additional stack usage.
This applies to SP1 and to any future backend equally.

### 6.2 Print-string context initialization

The `gfx_print_ctx_t` structs (`title_ctx`, `lives_display_ctx`,
`debug_ctx`) are currently initialized with SP1-specific struct
literals:

```c
struct sp1_pss title_ctx = {
    &title_area, SP1_PSSFLAG_INVALIDATE, 0, 0, 0, DEFAULT_BG_ATTR, 0, 0
};
```

For the abstraction, provide a macro or initializer function:

```c
#define GFX_PRINT_CTX_INIT(area, attr)  { &(area), GFX_PSS_INVALIDATE, 0, 0, 0, (attr), 0, 0 }
```

Each backend will define this macro to match its struct layout.

### 6.3 Sprite graphic data format

SP1 expects sprite graphics in a specific interleaved mask+pixel format
(SP1_TYPE_2BYTE, columns laid out sequentially).  This format is baked
into the sprite data at build time by `datagen.pl`.

The abstraction does NOT attempt to abstract the sprite graphic data
format — that is a concern for `datagen.pl` and whatever tool generates
the sprite data.  When integrating an alternative backend, the data
generation tool will need to output graphics in the format expected by
that backend.

### 6.4 SP1 callback for color setting

The `sprite_set_cell_attributes()` callback and the global
`sprite_attr_param` struct are SP1-specific mechanisms.  In the
abstraction, `gfx_sprite_set_color()` hides this entirely.  The
callback and its globals stay in `sprite.c` (guarded by
`#ifdef BUILD_FEATURE_SPRITE_ENGINE_SP1`) and are not exposed in the
public API.

### 6.5 `sprite_allocate()` → `gfx_sprite_create()`

The current `sprite_allocate()` in `sprite.c` contains SP1-specific
multi-step logic (the column-by-column creation pattern with
MASK2LB/MASK2/MASK2RB).  This is one of the few `gfx_*` functions that
cannot be a simple macro — it stays as a real function.

The function should be **renamed** to `gfx_sprite_create()` in
`sprite.c` and its body kept as-is for the SP1 backend.  When an
alternative backend is added, the body would be replaced with the
equivalent multi-step creation for that library (guarded by
`#ifdef BUILD_FEATURE_SPRITE_ENGINE_*`), or the function could be split
into backend-specific source files compiled conditionally.

The same applies to `gfx_sprite_set_color()` (currently
`sprite_set_color()`) and `gfx_init()` (currently `init_sp1()` in
`sp1engine.c`), which are also multi-step functions that stay as real
functions in their existing source files.

### 6.6 `sp1_Rect` in generated data

`datagen.pl` generates `struct sp1_Rect` definitions for game areas.
This must be changed to `gfx_rect_t`.  The struct layout (row, col,
width, height) is simple enough that any alternative backend should use
the same layout — if not, the `gfx_rect_t` typedef handles it.

### 6.7 Backward compatibility

Games that have custom code (`game_src/`) directly using SP1 functions
will need to be updated.  However, since the `gfx.h` header still
includes `<games/sp1.h>` for the SP1 backend, existing custom code will
still compile during a transition period.  A deprecation warning could
be added if desired.

## 7. Implementation order summary

1. **Create `gfx.h` + `gfx_sp1.h`** — generic API header + SP1 backend macros
2. **Update headers** — replace SP1 types with gfx typedefs
3. **Update engine sources** — replace SP1 calls with gfx calls (one
   file at a time, building and testing after each)
4. **Update banked code** — remove unnecessary SP1 includes
5. **Update `datagen.pl`** — change generated type names
6. **Update Makefile** — no source changes needed (macros only), verify build
7. **Full validation** — `make all-test-builds`, emulator testing

The recommended order for step 3 is:
1. `sp1engine.c` (init — foundational)
2. `screen.c` (global rect — used everywhere)
3. `sprite.c` (sprite lifecycle — used by hero, enemy, bullet)
4. `btile.c` (tile drawing — used by map, inventory, hero)
5. `charset.c` (tile registration — standalone)
6. `map.c` (screen drawing — uses sprites + tiles + text)
7. `hero.c` (hero drawing — uses sprites + tiles + text)
8. `enemy.c` (enemy drawing — uses sprites)
9. `bullet.c` (bullet drawing — uses sprites)
10. `debug.c` (debug output — uses text)
11. `inventory.c` (inventory display — uses tiles)
12. `game_loop.c` (heartbeat + update — uses tiles + update)

Each file can be migrated, compiled, and tested independently.
