# Tasks

## Task 1: Plan for cross-platform RAGE1 (ZX + CPC)

### Goal

Produce a detailed, phased evolution plan that turns RAGE1 from a ZX-only
engine into a cross-platform engine, allowing the same game — with mostly
shared `.gdata` plus minimal platform-specific overlays — to build for both
ZX Spectrum and Amstrad CPC.

The deliverable is documentation only; no engine refactor or code change
happens under Task 1 itself. Execution is a separate future task.

### Phase 1 scope (the plan must cover these in depth)

- **ZX Spectrum**: ZX48 and ZX128
- **Amstrad CPC**: CPC464, CPC664, CPC6128

### Out of Phase 1 (sketch only, no deep analysis)

- **MSX** and **Commodore 64** — note as long-horizon future directions
  only. Enough analysis to avoid architectural choices that would block
  them; not enough to actually plan their implementation. C64 in
  particular is a 6502 toolchain track and is explicitly out for now.

### Architectural anchors (decided; the plan reflects these, does not re-analyse them)

- **Asset model**: shared-core `.gdata` + per-platform overlays in a
  parallel tree (e.g. `game_data/<platform>/` or extended
  `patches/<platform>/`). Mostly-shared assets auto-convert at build
  time; per-platform overrides take precedence per file when divergence
  is irreducible. Plan defines exact precedence/merging rules. Existing
  `patches/` machinery is the natural precedent to extend.
- **HAL position**: the existing `gfx_*` API (introduced for the SP1↔JSP
  split, stable post-JSP closure) generalises into *the* multi-platform
  graphics HAL. SP1, JSP and a new CPC backend all sit behind the same
  surface. No second layer of abstraction above it. The plan starts with
  a `gfx_*` API audit for ZX-derived assumptions (cell coords, attribute
  model, BAT, screen dimensions).
- **`audio_*` HAL**: same shape as `gfx_*` — shared API surface, per-
  platform backends (ZX beeper, ZX AY, CPC AY). Music + SFX in scope.
- **`input_*` HAL**: same shape — gameplay-level events (up/down/fire/
  action keys) routed through a per-platform keyboard/joystick driver.
- **CPC graphics backend**: vendor an existing CPC library (cpctelera is
  the leading candidate, after a light comparative survey) rather than
  writing a new owned library. Almost certainly added as a git submodule
  under `external/` like JSP. Plan covers library selection, licence
  check, toolchain marriage, asset-pipeline integration.
- **Backwards-compatibility for existing ZX games**: best-effort. Each
  *phase* (not each commit) must end with `make all-test-builds` green
  and `tests/00regression/` screenshot tests green for ZX games.
  Mid-phase regressions are acceptable if they unblock the architecture,
  provided the phase-exit criterion restores green.

### Subsystems the plan must cover

- **Graphics rendering** — `gfx_*` API audit; CPC backend wrapping
  cpctelera; sprite/tile model generalisation
- **Audio** — `audio_*` HAL design; ZX (beeper + AY) and CPC (AY) backends;
  music + SFX format strategy
- **Input** — `input_*` HAL design; per-platform keyboard scanning and
  joystick handling
- **Asset pipeline** — shared core + overlays; `datagen.pl` /
  `mapgen.pl` / `btilegen.pl` changes; per-platform asset precedence
  rules; per-platform asset converters
- **Toolchain** — z88dk-only vs z88dk + SDCC; cpctelera integration;
  build-matrix naming (e.g. `PLATFORM=zx48|zx128|cpc464|cpc6128`); per-
  platform Makefile structure
- **CPC renderer library** — cpctelera evaluation, alternatives survey,
  integration approach, asset-tool integration
- **Banking / >64K memory** — ZX 128 paging vs CPC RAM expansion (CPC664/
  6128 has 128K via different mechanism); impact on RAGE1 datasets /
  codesets / SUBs, which currently assume ZX 0xC000-page semantics
- **Loaders and output packaging** — TAP/TZX (ZX), CDT/DSK (CPC); per-
  platform BASIC loader (Sinclair vs Locomotive); tape-vs-disk
  distribution
- **Testing** — per-platform emulator strategy (FUSE/JNEXT for ZX;
  Caprice32/ACE/RVM for CPC); extending `tests/00regression/`; CI matrix
  — the `rage1-z88dk` Docker image will need CPC tooling
- **Higher engine concepts** that currently assume ZX geometry — BTiles
  cell model (8×8 + attribute), hotzones (cell-based), screen dimensions
  (32×24 vs CPC mode-1 40×25). Review each: portable, parametrise, or
  platform-specific.

### Deliverable

A new directory **`doc/multiplatform-plan/`** containing:

- `README.md` — top-level: goal, scope, phasing across all subsystems,
  cross-cutting progress tracking, consolidated risks/open-questions
  index, links to per-subsystem docs
- `gfx.md` — graphics HAL audit + CPC backend integration
- `audio.md` — audio HAL design + per-platform backends
- `input.md` — input HAL design + per-platform backends
- `assets.md` — shared-core/overlay pipeline; datagen/mapgen/btilegen
  changes; per-platform asset precedence rules
- `toolchain.md` — z88dk/SDCC, cpctelera integration, build matrix
- `cpc-renderer.md` — cpctelera evaluation, alternatives, integration
  plan
- `banking.md` — >64K memory: ZX paging vs CPC expansion; dataset /
  codeset / SUB impact
- `testing.md` — per-platform emulators, regression framework extension,
  CI matrix

(File set is subject to small refinement during analysis — light files
may collapse, heavy ones may split.)

Each per-subsystem file must contain:

- Phased, step-by-step changes
- Each step individually testable where reasonable; phase-exit testing
  criteria explicit
- A **Risks** section
- An **Open Questions** section (for user to resolve before/during
  execution)

### Acceptance

Task 1 is complete when:

- All the docs above exist with phased work and the required sections
- Each phased step is described in enough detail to act on, or has an
  explicit unknown noted
- The user has reviewed and approved the **initial** plan

**The plan is a living document during execution.** Subsequent execution
tasks may revise phases, add or split steps, fold in risks discovered
during work, or correct architectural choices that don't survive contact
with reality. Task 1 closes when the initial draft is reviewed; ongoing
plan maintenance is part of execution, not a precondition for closing
Task 1.

### Process constraints

- Use **ultrathink** mode for this task and all of its subtasks
- Launch parallel agents whenever possible — up to **4 concurrent
  agents**. Per-subsystem deep analysis is naturally parallelisable (one
  agent per subsystem doc). The top-level `README.md` and any cross-
  subsystem consistency review are sequential and run **after** the
  subsystem drafts exist.
- Source-of-truth code analysis must be against the **current** repo,
  not recalled state — read files fresh
- Plan execution constraints (small commits, testable steps, commit
  message conventions, etc.) live inside the resulting plan, not in
  this task spec
- Launch independent reviewers for each task, and rework the task if the reviewer does not agree
